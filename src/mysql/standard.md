## 引言

> 写了三年 SQL，你知道什么时候该用 JOIN、什么时候该拆成多个查询吗？知道 `SELECT *` 到底损失了多少性能吗？

MySQL 是大多数业务系统的核心组件，但很多开发者只是"能用就行"，没有深入理解规范使用 MySQL 的最佳实践。一条不规范的 SQL，在测试环境跑得好好的，上了生产遇到大数据量就变成慢查询；一个不合理的索引设计，不仅没有加速查询，反而拖慢了写入性能。

本文整理了 **17 条 MySQL 使用规约**，每一条都来自真实生产环境的经验总结。涵盖 SQL 编写、索引设计、表结构规范三大维度。读完你将获得：

- 直接可以落地的 SQL 编写规范
- 索引设计的核心原则和避坑要点
- 表结构设计的最佳实践

## 一、SQL 编写规范

### 1.1 禁止使用 `SELECT *`

阿里开发规范明确指出：**禁止使用 `SELECT *`**。

**原因分析：**

| 问题 | 说明 |
|------|------|
| 字段变更导致异常 | 表中字段变更后，SQL 必须同步修改，否则执行可能出错 |
| 额外磁盘 I/O | 查询非必要的字段，徒增磁盘 I/O |
| 网络传输浪费 | 多余的字段数据增加网络传输量 |
| 无法使用覆盖索引 | `SELECT *` 必须回表，无法享受覆盖索引优化 |

**推荐写法：**

```sql
-- 明确指定需要的字段
SELECT id, name, status FROM user WHERE id = 1;
```

### 1.2 用小表驱动大表

关联查询时，**先用小表查询结果，再用结果去大表查询**，可以大大减少连接次数。

```sql
-- 部门数量远小于员工数量，部门表作为驱动表
SELECT * FROM department
INNER JOIN employee ON department.id = employee.department_id
WHERE department_name = '部门1';
```

> **💡 核心提示**：MySQL 优化器通常会自动选择小表作为驱动表，但显式控制连接顺序（使用 `STRAIGHT_JOIN`）在某些复杂场景下更可靠。

### 1.3 JOIN 关联表不宜过多

| 规范 | 说明 |
|------|------|
| 关联表上限 | **不超过 3 张** |
| 关联字段类型 | 必须保持一致 |
| 索引要求 | 每张表的关联字段都要建索引 |

关联表过多会增加查询时间、产生临时表缓存中间结果。建议拆成多条小 SQL 在应用层组装。

### 1.4 禁止使用左模糊或全模糊查询

```sql
-- ❌ 左模糊查询（索引失效）
SELECT * FROM user WHERE name = '%一灯';

-- ❌ 全模糊查询（索引失效）
SELECT * FROM user WHERE name = '%一灯%';

-- ✅ 右模糊查询（可用索引）
SELECT * FROM user WHERE name = '一灯%';
```

根据 B+ 树的特性，左模糊和全模糊无法利用索引的有序性，只能全表扫描。

> **💡 核心提示**：如果确实需要全模糊搜索，考虑使用 **Elasticsearch** 或 MySQL 的 **FULLTEXT** 全文索引。

### 1.5 索引访问类型至少达到 range 级别

索引访问类型从好到差排列：

| 类型 | 说明 | 性能 |
|------|------|------|
| `system` | 表只有一行数据 | 最优 |
| `const` | 通过主键或唯一索引等值查询 | 极好 |
| `eq_ref` | JOIN 时使用主键或唯一索引 | 很好 |
| `ref` | 通过普通索引等值查询 | 好 |
| `range` | 索引范围扫描 | 可接受 |
| `index` | 全索引扫描 | 较差 |
| `ALL` | 全表扫描 | 最差 |

**要求：至少达到 `range` 级别，最好到 `const` 级别。**

### 1.6 更优雅地使用联合索引

#### 最左前缀原则

联合索引 `(b, c, a)` 的排序很重要。优先把**区分度高**的字段放在最左边。

统计字段区分度：

```sql
SELECT 
    COUNT(DISTINCT birthday) / COUNT(*), 
    COUNT(DISTINCT gender) / COUNT(*) 
FROM user;
```

值越大，区分度越高。

#### 经典面试题

> SQL：`SELECT a FROM table_name WHERE b = 1 ORDER BY c;`

**答案：联合索引顺序应该是 `(b, c, a)`。**

原因：
- `b` 在 WHERE 条件中，放最前面用于过滤
- `c` 在 ORDER BY 中，放在第二位用于排序
- `a` 在 SELECT 中，放在最后用于覆盖索引（避免回表）

#### SQL 执行顺序参考

```
FROM → ON → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

### 1.7 避免深分页

```sql
-- ❌ 深分页性能差
SELECT * FROM user WHERE name = '一灯' LIMIT 10000, 10;

-- ✅ 使用子查询减少回表
SELECT * FROM user 
WHERE id IN (
    SELECT id FROM user 
    WHERE name = '一灯' 
    LIMIT 10000, 10
);
```

> **💡 核心提示**：子查询方案的核心是**延迟关联**——先在覆盖索引中拿到主键 ID，再用 ID 回表查完整数据，大幅减少回表次数。

### 1.8 `IN` 条件数量不宜过多

`IN` 条件中的值**不要超过 1000 个**，否则：
- 解析 SQL 耗时增加
- 执行计划优化耗时增加
- 可能导致锁范围扩大

建议拆成多批次查询或使用临时表。

## 二、表结构设计规范

### 2.1 单表字段不要超过 30 个

字段数量过多会拖慢查询性能，加载大量无用字段浪费 I/O。

**超过 30 个字段时**：使用**垂直分表**，将冷热字段分离。

```mermaid

    A[原始大表 50 个字段] --> B[热数据表<br/>常用字段 15 个]
    A --> C[冷数据表<br/>不常用字段 35 个]
    B -->|通过主键关联| C
    
    style B fill:#ccffcc
    style C fill:#e8f4fd
```

### 2.2 枚举字段不要使用字符类型

| 类型 | 存储空间 | 推荐度 |
|------|---------|--------|
| `VARCHAR` | 变长，额外 1-2 字节记录长度 | 不推荐 |
| `TINYINT` | 1 字节 | 推荐 |
| `UNSIGNED TINYINT` | 1 字节，范围 0-255 | **最推荐** |

存储枚举值或布尔值（是否标志）时，使用 `TINYINT` 而不是 `VARCHAR`。

### 2.3 小数类型禁止使用 `FLOAT` 和 `DOUBLE`

`FLOAT` 和 `DOUBLE` 存在精度损失问题，无法得到精确的计算结果。

```sql
-- ❌ 精度损失
SELECT 0.1 + 0.2;  -- 可能得到 0.30000000000000004

-- ✅ 精确存储
-- 使用 DECIMAL(10, 2) 存储金额等精确小数
```

**涉及金额、价格等精确小数时，必须使用 `DECIMAL` 类型。**

### 2.4 所有字段必须设置默认值且不允许为 `NULL`

| 问题 | 说明 |
|------|------|
| 额外存储空间 | `NULL` 值需要额外的存储空间记录 NULL 标志位 |
| 索引不包含 NULL | MySQL 索引不索引 NULL 值，查询 `IS NULL` 无法使用索引 |
| 空指针异常 | 数值类型返回 NULL 时，映射到实体类可能报空指针 |

> **💡 核心提示**：给每个字段设置合理的默认值，如字符串设为 `''`，数值设为 `0`。

### 2.5 必须创建主键，最好是有序数值类型

- InnoDB 表**必须有主键**。如果不设置，InnoDB 会自动生成隐藏列作为主键，我们无法使用且占用更多存储。
- **自增主键**（`AUTO_INCREMENT`）最适合做 InnoDB 的主键：插入有序，不会频繁调整 B+ 树结构，避免页分裂。

### 2.6 禁止创建预留字段

| 问题 | 说明 |
|------|------|
| 语义不明确 | 无法通过字段名判断用途 |
| 类型可能不合适 | 预留的 `VARCHAR(255)` 可能实际需要 `INT` |
| 无法创建合适索引 | 不知道用途就无法决定是否需要索引 |

**需要扩展时，直接添加语义明确的新字段。**

### 2.7 单表索引数不要超过 5 个

| 索引数量 | 影响 |
|---------|------|
| 过少（0-2 个） | 查询效率低，全表扫描 |
| 适中（3-5 个） | **最佳平衡点** |
| 过多（6+ 个） | 写入性能下降，占用额外存储 |

索引提高查询效率，但会拖慢 `INSERT`/`UPDATE`/`DELETE` 性能（需要同时更新索引）。

## 三、操作规范

### 3.1 快速判断记录是否存在

```sql
-- ❌ 使用 COUNT（全表扫描，实时计算行数）
SELECT COUNT(*) FROM user WHERE name = '一灯';

-- ✅ 使用 LIMIT 1（匹配到一条就返回）
SELECT id FROM user WHERE name = '一灯' LIMIT 1;
```

InnoDB **没有缓存表总行数**，`COUNT(*)` 需要实时扫描，效率低。`LIMIT 1` 配合覆盖索引，效率更高。

### 3.2 修改操作之前先查询

```sql
-- ❌ 直接更新（可能锁大量行）
UPDATE `order` SET status = 'deleted' WHERE status = 'canceled';

-- ❌ 直接删除（可能误删大量数据）
DELETE FROM `order` WHERE status = 'deleted';

-- ✅ 先查后改（推荐）
SELECT id FROM `order` WHERE status = 'canceled';
UPDATE `order` SET status = 'deleted' WHERE id IN (?, ?, ?);

SELECT id FROM `order` WHERE status = 'deleted';
DELETE FROM `order` WHERE id IN (?, ?, ?);
```

**先查后改的好处：**

| 优势 | 说明 |
|------|------|
| 锁行更少 | 使用主键 ID 只锁定目标行 |
| 可以分批 | 控制每次处理的数量，避免大事务 |
| 可审计 | 日志中能记录具体更新了哪些数据 |
| 更安全 | 降低误操作的风险 |

## 四、生产环境避坑指南

1. **`SELECT *` 是性能杀手**：不仅浪费 I/O 和网络带宽，还会导致覆盖索引失效。养成显式指定字段的好习惯。
2. **JOIN 超过 3 张表立即报警**：在生产环境中，超过 3 张表的 JOIN 几乎总是性能问题的根源。应该在代码层拆分查询。
3. **`NULL` 值的隐性成本**：`NULL` 不仅多占存储空间，还影响索引效率和 `COUNT` 统计结果。所有字段都应有默认值。
4. **联合索引不是越多越好**：一个 `(a, b, c)` 的联合索引等价于 `(a)`、`(a, b)`、`(a, b, c)` 三个索引。避免重复创建重叠的联合索引。
5. **`DECIMAL` 的精度陷阱**：`DECIMAL(10, 2)` 最大只能存 99999999.99（8 位整数 + 2 位小数），设计金额字段时确保整数部分够用。
6. **深分页必须优化**：`LIMIT 100000, 10` 在生产环境是不可接受的。使用游标分页或延迟关联方案。
7. **主键选择影响全局性能**：无序主键（如 UUID）会导致大量页分裂和随机 I/O，InnoDB 表必须使用自增主键或雪花算法生成的有序 ID。

### 5.2 行动清单

1. **代码审查**：检查项目中的 SQL 是否存在 `SELECT *`、多表 JOIN（>3 张）、`LIKE '%xxx%'` 等违规写法。
2. **慢查询分析**：开启慢查询日志（`long_query_time = 1`），定期分析并优化慢 SQL。
3. **索引审查**：使用 `SHOW INDEX FROM table_name` 检查每张表的索引数量和设计合理性。
4. **表结构审查**：检查是否存在 `NULL` 字段、预留字段、字段超过 30 个的宽表。
5. **制定团队规范**：将这 17 条规约整理为团队的 SQL 开发规范文档，纳入 Code Review 检查项。
6. **使用 SQL 审查工具**：集成 SOAR、Archery 等 SQL 审核工具，在 CI/CD 流程中自动拦截不合规 SQL。
7. **定期培训**：每季度组织一次 SQL 性能优化分享，持续提高团队的数据库使用水平。
